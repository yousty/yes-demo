# TaskFlow ‚Äî `yes` event-sourcing demo

A small, fully-dockerised Kanban-style task tracker built as a showcase for the
[`yes`](https://github.com/yousty/yes) event-sourcing framework. The backend is
Rails 8.1 + `yes-core` / `yes-command-api` / `yes-read-api` backed by
[`pg_eventstore`](https://github.com/yousty/pg_eventstore); the frontend is
Next.js 16 (App Router) and talks to the backend **only** through the command
and read APIs.

## Domain

A single bounded context, `TaskFlow`, with two aggregates:

### `TaskFlow::Board::Aggregate` (`read_model :board`)

A Kanban board with an owner, members, and a set of tasks.

| Command | Type | Notable invariants (guards) |
|---|---|---|
| `create_board` | custom (explicit `event :created`) | `no_change` if already created, `title_present` |
| `change_title` | shortcut `command :change, :title` | default `no_change` |
| `change_description` | custom (nullable payload) | `no_change` |
| `archive` | custom (`event :archived`) ‚Äî **owner only** | `no_change` if already archived |
| `unarchive` | custom (`event :unarchived`) ‚Äî **owner only** | `no_change` if not archived |
| `add_member` | custom | `not_archived`, `no_change` if already member |
| `remove_member` | custom ‚Äî **owner only** | `not_owner`, `is_member` |
| `transfer_ownership` | custom (`event :ownership_transferred`) ‚Äî **owner only** | `different_owner`, `new_owner_is_member` |
| `remove` | generated by `removable` | default |

### `TaskFlow::Task::Aggregate` (`read_model :task`, `parent :board`)

A single task on a board. Access is gated by a board-membership authorizer.

| Command | Type | Notable invariants |
|---|---|---|
| `create_task` | custom (`event :created`) | `board_exists`, `board_not_archived`, `no_change` if already created |
| `change_title` | shortcut | default |
| `change_description` | custom (nullable) | `no_change` |
| `change_priority` | shortcut with custom `:task_priority` enum | default |
| `set_due_date` | custom (`event :due_date_set`, nullable date) | `no_change` |
| `assign_to_member` | custom (`event :member_assigned`) | `assignee_is_board_member`, `board_not_archived` |
| `unassign` | custom | `currently_assigned` |
| `start` | custom | `status == todo`, `board_not_archived` |
| `complete` | custom | `status == in_progress` |
| `reopen` | custom | `status == done` |
| `cancel` | custom | `status not in [done, cancelled]` |
| `remove` | generated by `removable` | default |

Both aggregates demonstrate:

- **Command shortcuts** (`command :change, :title`, `command :change, :priority, :task_priority`).
- **Custom commands with explicit event names** for verbs the resolver doesn‚Äôt know.
- **Small, targeted payloads** (no big edit-everything commands).
- **Inline aggregate- and per-command authorizers** using `auth_data[:identity_id]`.
- **Guards that read across aggregates** (e.g. the task's `assign_to_member` consults the board‚Äôs `member_ids`).
- **Custom types** (`:task_status`, `:task_priority`, `:date`) registered in `config/initializers/yes_types.rb`.
- **`removable` helper** for soft-delete with `removed_at`.
- **`parent :board, command: false`** so the parent link is recorded in state but the create flow bundles `board_id` into a single call.

## Frontend

- **Next.js 16** with the App Router, React 19, and server components.
- Three demo users (Alice, Bob, Carol) are baked into
  `frontend/lib/users.ts`. A top-bar `UserSwitcher` swaps the signed-in user by
  writing a cookie; every server component reads the cookie and mints the dev
  bearer token (Base64-encoded `{identity_id, user_id}`) before calling the
  backend.
- All mutations go through the **command API** via server actions in
  `frontend/app/_actions/`; all reads go through the **read API** via
  `frontend/lib/api.ts`. No direct database access and no REST.
- Pages:
  - `/` ‚Äî grid of boards for the current user + a _new board_ form.
  - `/boards/[id]` ‚Äî four-column Kanban with per-task transition buttons, a
    member side panel, and an owner-only danger zone (archive, transfer
    ownership).

## Console usage

The Rails console is the fastest way to drive aggregates, inspect read
models, and look at the underlying event stream. Open one inside the
backend container:

```bash
docker compose exec backend bundle exec rails console
```

### Aggregate shortcuts

`config/initializers/yes_core.rb` sets `config.aggregate_shortcuts = true`,
so on console boot `yes-core` discovers every aggregate under
`app/contexts/**/aggregate.rb` and registers a short-hand constant for it.
Type `shortcuts` to list them:

```
> shortcuts

Available Aggregate Shortcuts:
======================================================================
TF::Board ‚Üí TaskFlow::Board::Aggregate
TF::Task  ‚Üí TaskFlow::Task::Aggregate
======================================================================
```

`TF` is a freshly-built shortcut module (not an alias of the real
`TaskFlow` namespace), so `TF::Board.new(id)` returns an aggregate
instance and `TaskFlow::Board` still resolves to the namespace module
that contains the Aggregate class.

### Running commands

Each `command :name` declared in an aggregate becomes an instance method.
Payload keys map 1-to-1 to the `payload` declaration in the command. Use
the demo identities from `frontend/lib/users.ts` so you can see the
results in the UI immediately:

```ruby
require 'securerandom'

ALICE = '11111111-1111-4111-8111-111111111111'
BOB   = '22222222-2222-4222-8222-222222222222'

board_id = SecureRandom.uuid
board    = TF::Board.new(board_id)
board.create_board(title: 'Sprint 42', description: 'Q2 goals', owner_id: ALICE)
board.add_member(member_id: BOB)
board.change_title(title: 'Sprint 42 ‚Äî revised')

task_id = SecureRandom.uuid
task    = TF::Task.new(task_id)
task.create_task(board_id: board_id, title: 'Wire up payments', priority: 'high')
task.assign_to_member(assignee_id: BOB)
task.start
task.complete
```

Each call returns a `Yes::Core::Commands::Response`; check
`response.success?` and `response.event` if you need the emitted event.

> **Authorization.** Direct aggregate calls bypass the command API's
> per-command authorizer ‚Äî your console session is implicitly
> super-admin. To exercise the `authorize do ‚Ä¶ end` block (e.g. "must be
> a member of this board"), drive the aggregates through `POST
> /v1/commands` instead, as shown in the curl smoke test above.

### Inspecting commands and state

```ruby
TF::Board.new.commands
# => { add_member: [:member_added], archive: [:archived],
#      change_description: [:description_changed], ... }

board.title          # => "Sprint 42 ‚Äî revised"
board.member_ids     # => ["11111111-...", "22222222-..."]
board.archived       # => false
board.reload         # re-fetch the read model after an out-of-band change
```

Read models are plain ActiveRecord ‚Äî `TF::Board` writes into the `boards`
table (`Board` AR class), `TF::Task` writes into `tasks` (`Task` AR
class). Either go through the aggregate or query AR directly:

```ruby
Board.find(board_id).title
Task.where(board_id: board_id, status: 'in_progress').pluck(:title, :assignee_id)
```

### Inspecting the event stream

```ruby
board.events.to_a.map(&:type)
# => ["TaskFlow::BoardCreated", "TaskFlow::Board::MemberAdded",
#     "TaskFlow::Board::TitleChanged"]

board.latest_event   # last PgEventstore::Event in the stream
board.event_revision # zero-based stream revision of that event
```

For a graphical view of the same stream open
http://localhost:3100/admin/eventstore/ and search for
`TaskFlow-Board-<board_id>`.

## What is deliberately *not* here

- **No subscriptions worker.** `Yes::Core.process_commands_inline = true`, so the
  command handler updates the local read models synchronously. A separate
  `pg-eventstore subscriptions start` process would only be necessary if another
  service needed to consume the event stream.
- **No real auth.** `DevAuthAdapter` accepts any Base64-encoded JSON token. The
  frontend hardcodes three demo identities so you can exercise per-user
  authorization (e.g. only the owner can archive a board).
- **No CI/CD, no observability wiring.** The goal is to keep the demo
  approachable; wiring Sentry, OpenTelemetry, or Cerbos would mirror the
  patterns in `yes-core`‚Äôs README.
