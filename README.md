# TaskFlow — `yes` event-sourcing demo

A small, fully-dockerised Kanban-style task tracker built as a showcase for the
[`yes`](https://github.com/yousty/yes) event-sourcing framework. The backend is
Rails 8.1 + `yes-core` / `yes-command-api` / `yes-read-api` backed by
[`pg_eventstore`](https://github.com/yousty/pg_eventstore); the frontend is
Next.js 16 (App Router) and talks to the backend **only** through the command
and read APIs.

## Domain

A single bounded context, `TaskFlow`, with two aggregates:

### `TaskFlow::Board::Aggregate` (`read_model :board`)

A Kanban board with an owner, members, and a set of tasks.

| Command | Type | Notable invariants (guards) |
|---|---|---|
| `create_board` | custom (explicit `event :created`) | `no_change` if already created, `title_present` |
| `change_title` | shortcut `command :change, :title` | default `no_change` |
| `change_description` | custom (nullable payload) | `no_change` |
| `archive` | custom (`event :archived`) — **owner only** | `no_change` if already archived |
| `unarchive` | custom (`event :unarchived`) — **owner only** | `no_change` if not archived |
| `add_member` | custom | `not_archived`, `no_change` if already member |
| `remove_member` | custom — **owner only** | `not_owner`, `is_member` |
| `transfer_ownership` | custom (`event :ownership_transferred`) — **owner only** | `different_owner`, `new_owner_is_member` |
| `remove` | generated by `removable` | default |

### `TaskFlow::Task::Aggregate` (`read_model :task`, `parent :board`)

A single task on a board. Access is gated by a board-membership authorizer.

| Command | Type | Notable invariants |
|---|---|---|
| `create_task` | custom (`event :created`) | `board_exists`, `board_not_archived`, `no_change` if already created |
| `change_title` | shortcut | default |
| `change_description` | custom (nullable) | `no_change` |
| `change_priority` | shortcut with custom `:task_priority` enum | default |
| `set_due_date` | custom (`event :due_date_set`, nullable date) | `no_change` |
| `assign_to_member` | custom (`event :member_assigned`) | `assignee_is_board_member`, `board_not_archived` |
| `unassign` | custom | `currently_assigned` |
| `start` | custom | `status == todo`, `board_not_archived` |
| `complete` | custom | `status == in_progress` |
| `reopen` | custom | `status == done` |
| `cancel` | custom | `status not in [done, cancelled]` |
| `remove` | generated by `removable` | default |

Both aggregates demonstrate:

- **Command shortcuts** (`command :change, :title`, `command :change, :priority, :task_priority`).
- **Custom commands with explicit event names** for verbs the resolver doesn’t know.
- **Small, targeted payloads** (no big edit-everything commands).
- **Inline aggregate- and per-command authorizers** using `auth_data[:identity_id]`.
- **Guards that read across aggregates** (e.g. the task's `assign_to_member` consults the board’s `member_ids`).
- **Custom types** (`:task_status`, `:task_priority`, `:date`) registered in `config/initializers/yes_types.rb`.
- **`removable` helper** for soft-delete with `removed_at`.
- **`parent :board, command: false`** so the parent link is recorded in state but the create flow bundles `board_id` into a single call.

## Frontend

- **Next.js 16** with the App Router, React 19, and server components.
- Three demo users (Alice, Bob, Carol) are baked into
  `frontend/lib/users.ts`. A top-bar `UserSwitcher` swaps the signed-in user by
  writing a cookie; every server component reads the cookie and mints the dev
  bearer token (Base64-encoded `{identity_id, user_id}`) before calling the
  backend.
- All mutations go through the **command API** via server actions in
  `frontend/app/_actions/`; all reads go through the **read API** via
  `frontend/lib/api.ts`. No direct database access and no REST.
- Pages:
  - `/` — grid of boards for the current user + a _new board_ form.
  - `/boards/[id]` — four-column Kanban with per-task transition buttons, a
    member side panel, and an owner-only danger zone (archive, transfer
    ownership).

## Manual smoke-test via curl

```bash
TOKEN=$(echo -n '{"identity_id":"11111111-1111-4111-8111-111111111111","user_id":"11111111-1111-4111-8111-111111111111"}' | base64)
BOARD_ID=$(uuidgen | tr 'A-Z' 'a-z')

curl -sS -X POST http://localhost:3100/v1/commands \
  -H "authorization: Bearer $TOKEN" \
  -H "content-type: application/json" \
  -d "{\"commands\":[{\"context\":\"TaskFlow\",\"subject\":\"Board\",\"command\":\"CreateBoard\",\"data\":{\"board_id\":\"$BOARD_ID\",\"title\":\"Sprint 1\",\"description\":\"First sprint\",\"owner_id\":\"11111111-1111-4111-8111-111111111111\"}}]}"

curl -sS -H "authorization: Bearer $TOKEN" "http://localhost:3100/v1/queries/boards"
```

Then visit http://localhost:3100/admin/eventstore/ and look at the
`TaskFlow-Board-<id>` stream to see the emitted `TaskFlow::BoardCreated`
event.

## Development

### Running the aggregate specs

All aggregate invariants are covered by specs written with the `yes` test DSL
(`success` / `invalid` / `no_change` blocks, plus `have_authorizer`,
`have_read_model_class`, `have_parent` matchers):

```bash
docker compose exec backend bundle exec rspec
```

> **117 examples, 0 failures.**

### Running rubocop

```bash
docker compose exec backend bundle exec rubocop
```

### Regenerating read-model migrations

If you add or change an aggregate attribute, re-run:

```bash
docker compose exec backend bundle exec rails generate yes:core:read_models:update
docker compose exec backend bundle exec rails db:migrate
```

## What is deliberately *not* here

- **No subscriptions worker.** `Yes::Core.process_commands_inline = true`, so the
  command handler updates the local read models synchronously. A separate
  `pg-eventstore subscriptions start` process would only be necessary if another
  service needed to consume the event stream.
- **No real auth.** `DevAuthAdapter` accepts any Base64-encoded JSON token. The
  frontend hardcodes three demo identities so you can exercise per-user
  authorization (e.g. only the owner can archive a board).
- **No CI/CD, no observability wiring.** The goal is to keep the demo
  approachable; wiring Sentry, OpenTelemetry, or Cerbos would mirror the
  patterns in `yes-core`’s README.
